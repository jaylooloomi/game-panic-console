## 1. 目標（這份切片要回答的唯一問題）

> 「被強迫每 20 秒在多個**同時運行**的小遊戲之間切換，是『緊張到上癮』，還是『煩躁到想關掉』？」

這是整個專案最大的賭注。切片用**最低成本**驗證核心樂趣，**不碰**連線、真串流、Steam、角色、卡牌、商業化。

## 2. 範圍

**做（In Scope）**
- 3 個同時運行的小遊戲，刻意選三種互斥的大腦模式（反射 / 空間 / 節奏）。
- 20 秒強制切換 + 倒數 3 秒紅色警報。
- 共享 HP（預設 5）、生存計時、分數。
- 可即時調整切換間隔的 debug HUD。
- 單機、單一玩家。

**不做（Out of Scope，留待後續階段）**
- 1v1 連線、串流整合、Steam/Steamworks。
- 卡牌 / 能量系統、英雄角色。
- 其餘 12+ 小遊戲。
- 商業化、正式美術與音效、CRT/Glitch 後處理（切片可極簡或省略）。

## 3. 成功標準

切片視為「通過」當且僅當：
1. 試玩者出現明顯的張力與心流，且主動想「再來一局」。
2. **旁觀者光看畫面就能感受到混亂感**（對直播至關重要）。
3. 三個小遊戲的切換讓人「腦袋要重新開機」，而非單純看不懂。

未通過時的處置：調整切換秒數 / 小遊戲組合 / 背景行為，再測；若反覆無法成立，則在進一步投資前重新檢視核心概念。

## 4. 架構與元件

**核心設計原則**：模擬（simulation）與「輸入 / 渲染 / 焦點」徹底解耦。每個小遊戲**每一幀都在跑**；「焦點」只決定三件事：(a) 是否讀取玩家輸入、(b) 用前台正常或背景慢速模擬、(c) 是否顯示於主螢幕。

### 4.1 `MinigameBase`（抽象類別）— 全專案地基
每個小遊戲的合約。之後的 15+ 小遊戲與玩家自製（JS）遊戲都實作同一合約。

```csharp
public abstract class MinigameBase : MonoBehaviour
{
    public string GameId { get; protected set; }
    public event System.Action<MinigameBase> OnFail;   // 觸發失敗時通知 GameSession

    public abstract void Init();                       // 一次性建立
    public abstract void ResetGame();                  // 重置到起始狀態
    public abstract void OnFocus();                    // 成為前台（開始讀輸入、正常速度）
    public abstract void OnBlur();                     // 退到背景（停讀輸入、進入背景行為）
    public abstract void Tick(float dt, bool isFocused); // 每幀推進；不論前後台都呼叫

    protected void TriggerFail() => OnFail?.Invoke(this);
}
```

建議每個小遊戲把「核心模擬」寫成**可獨立測試的純 C# 類別**，MonoBehaviour 僅作外殼（持有純邏輯 + 負責渲染/輸入橋接）。

### 4.2 `SwitchController`（核心循環）
- 持有目前 active 的小遊戲清單（切片為 3 個）與焦點索引。
- 管理切換計時器、倒數 3 秒警報（音效 + 前台螢幕抖動/雜訊）。
- 計時歸零 → `SwitchFocus()`：對舊前台呼叫 `OnBlur()`、對新前台呼叫 `OnFocus()`，並依曲線重設計時。
- **每幀對所有小遊戲呼叫 `Tick(dt, isFocused)`**（這是背景遊戲持續運作的關鍵）。
- 切換間隔曲線可由 debug HUD 即時調整。

### 4.3 `GameSession`（對局狀態）
- 持有共享 HP（預設 5）、生存計時、分數、是否 Game Over。
- 訂閱每個小遊戲的 `OnFail` → HP - 1 → 對該小遊戲 `ResetGame()` → 若 HP = 0 觸發 Game Over。

### 4.4 `InputRouter`
- 只把玩家原始輸入轉發給**前台**小遊戲。
- **未來「按鍵反轉 / 禁用輸入」卡牌的插入點**（切片不實作，但介面保留可包裝輸入的縫）。

### 4.5 渲染 / UI（切片版，刻意極簡）
- 每個小遊戲各自一台 Camera + RenderTexture。
- 控制台 UI：前台大畫面 + 另兩個即時縮圖（呼應主視覺）。
- 控制台模型用 2D 假面板即可；3D 控制台與後處理特效屬後續。
- Debug HUD：切換間隔滑桿、重新開始、顯示 HP / 計時 / 分數。

## 5. 資料流（每幀）

1. `SwitchController.Update()`：扣減計時器。
2. 計時器 ≤ 3 秒 → 觸發警報（音效 + 前台螢幕抖動）。
3. 計時器 ≤ 0 → `SwitchFocus()`（舊前台 `OnBlur` / 新前台 `OnFocus` / 依曲線重設計時）。
4. 對每個小遊戲呼叫 `Tick(dt, isFocused)`。
5. `InputRouter` 讀輸入 → 僅轉發給前台小遊戲。
6. 任一小遊戲 `TriggerFail()` → `GameSession`：HP - 1、`ResetGame()`；HP = 0 → Game Over。

## 6. 數值預設（提案值，可於試玩時微調）

| 參數 | 預設 |
|------|------|
| 共享 HP | 5 |
| 切換間隔曲線 | 第 1–3 輪 20 秒；第 4–6 輪 15 秒；第 7 輪起 10 秒 |
| 切換警報 | 倒數 3 秒起 |
| 並行小遊戲數 | 3 |
| 重新聚焦預警 | 0.5 秒解凍緩衝 |

## 7. 三個小遊戲規格

### 7.1 像素小恐龍 Dino（反射神經 · 1 鍵）
- 玩法：地面自動捲動，障礙出現，玩家按跳躲避。
- 失敗：撞到障礙 → `TriggerFail()`。
- `OnBlur`：完全凍結。
- `OnFocus`：0.5 秒預警後恢復，保留當前捲動速度。

### 7.2 貪食蛇 Snake（空間規劃 · 方向鍵）
- 玩法：格狀盤面（建議 15×15），蛇朝方向鍵移動吃食物變長。
- 失敗：撞牆或撞自己 → `TriggerFail()`。
- `OnBlur`：以 25% 速度沿當前方向直線前進（不轉向），避免切回即撞牆。
- `OnFocus`：0.5 秒預警後恢復正常速度。

### 7.3 別踩白塊 Piano Tiles（節奏精準 · 多鍵連點）
- 玩法：4 直行，黑塊由上往下落，玩家在對應行按鍵點掉。
- 失敗：黑塊落到底未被點，或點到空白行 → `TriggerFail()`。
- `OnBlur`：黑塊停止下落（凍結）。
- `OnFocus`：0.5 秒預警後恢復下落。

> 選這三款的理由：三種**完全不同的操作與思考模式**，讓「切換」本身產生認知成本——這正是遊戲賣點。三者皆便宜可做，且各有清楚的背景行為。

## 8. 為未來鋪路（不在切片實作，但縫先留好）

- **卡牌**：未來以 `CardEffect` 掛在 `SwitchController` / `GameSession` / `InputRouter` 開的 hook 上。
- **串流**：觀眾指令對應到與卡牌**同一組**效果觸發點。
- **連線**：`GameSession` 之後成為同步狀態；小遊戲模擬需對連線友善（事件同步或近似決定論）——切片階段僅記錄此約束，不解。
- **玩家自製（JS）**：UGC 遊戲即「`Tick` 內呼叫 Jint 執行玩家 JS」的 `MinigameBase` 子類。

## 9. 測試策略

- **EditMode 單元測試（純邏輯，不需畫面）**：
  - 經過設定秒數後焦點確實切換。
  - 小遊戲 `TriggerFail()` 會使 HP - 1 並 `ResetGame()`。
  - HP = 0 觸發 Game Over。
  - 背景小遊戲確實每幀被 `Tick`（背景行為有推進）。
  - 切換間隔曲線（20→15→10）依輪數正確套用。
- **小遊戲核心模擬**寫成純 C# 類別，便於測試與 AI 協助開發。
- **「好不好玩」**靠真人試玩（切片存在的唯一理由）。
